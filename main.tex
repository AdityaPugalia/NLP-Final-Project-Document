\section{Introduction}

\subsection{Motivation}

Large language Models have demonstrated strong performance across a wide range of natural language processing tasks, but training and deploying them typically requires substantial computational resources as some of the modern frontier systems reach hundreds of billions in parameters \parencite{deepseekai2025deepseekv3technicalreport}. This makes it difficult to experiment with them in a small-scale setting that can be trained and tested locally. \textit{NanoChat} \parencite{nanochat} addresses this issue by providing a minimal, end-to-end chatbot framework designed to reproduce GPT-2-style language modelling behaviour at a much smaller scale and cost. Its compact design makes it a useful experimental testbed: it is large enough to capture the essential components of modern autoregressive language models, yet small enough to be trained and modified under limited academic resources. As a result, NanoChat can be used as an ideal platform for systematically evaluating whether recent advances in architecture and training can produce meaningful gains in practice.

\subsection{Objective}

Through this project we aim to introduce \textbf{Nanochat 2.0}, an improved small scale language model built on top of the NanoChat framework. The main research objective is to determine \textit{to what extent can recent architectural and training innovations improve the efficiency and performance of a small-scale language model like NanoChat.} We chose this research topic because compact language models make experimentation more computationally accessible, while also providing a controlled setting to test whether the benefits of recent methods stem from the techniques themselves rather than from large model size. Moreover, improving such models is practically valuable because it allows the capabilities of autoregressive language modelling to be deployed in settings where full-scale LLMs are impractical, such as offline use, on-device inference on resource-constrained hardware such as mobile devices, or applications whose requirements do not justify the cost of a frontier-scale model.

\subsection{NLP Tasks}

Large Language Models are commonly associated with broad downstream capabilities such as summarisation, machine translation, question answering, and sentiment analysis. These abilities are fundamentally built on the underlying task of autoregressive language modelling. Accordingly, our project focuses primarily on improving NanoChat as an \textbf{autoregressive language model for open-ended text generation}, with the expectation that better language modelling performance will translate into stronger general-purpose NLP capability.

\subsection{Activation Functions}

We investigate substituting the canonical ReLU activation function with experimental activation functions recently discovered via evolutionary search \parencite{vitvitskyi2026mininggeneralizableactivationfunctions}, for instance, by perturbing the GeLU function to obtain GELUSine and GELU-Sinc. These activations were demonstrated to improve the performance on computer vision tasks with small models. As such, we investigate if their validity generalises to natural language processing.

In our experiments, we perform a baseline experiment on the canonical ReLU activation function. In addition, we experiment by substituting all activation functions in the neural network with the following activation functions on an all-at-once basis.

\begin{enumerate}
    \item ReLU\(^2\), i.e. \(\mathrm{ReLU}^2(x)\)
    \item Gaussian Error Linear Unit (GeLU) \parencite{hendrycks2016gelu}
        \[\mathrm{GeLU}(x)=\frac{x}{2}\left[1+\mathrm{erf}\left({\frac{x}{\sqrt{2}}\right)\right]\]
    \item GeLUSine
        \[\mathrm{GeLUSine}(x)= \mathrm{GeLU}(x)+0.1\sin{x}\]
    \item GeLU-Sinc-Perturbation
        \[\mathrm{GeLUSine}(x)= \mathrm{GeLU}(x)[1+\alpha\cdot\mathrm{sinc}(x)]
        =\mathrm{GeLU}(x)\left[1+\frac{\alpha\sin(\pi x)}{2\pi x}\right]\]
        \(\alpha=0.5\) in our experiments
    \item Gaussian-Modulated Tangent Unit (GMTU)
        \[f(x) = \underbrace{\alpha \tanh(\beta x) \cdot e^{-\gamma x^2}}_{\text{primary response}} + \underbrace{\lambda x}_{\text{linear leak}}\]
        \(\alpha=1.0, \beta=1.5, \gamma=0.2, \lambda=0.1\) in our experiments
    \item Turbulent\\
        Unlike other activation functions which are applied element-wise, the turbulent activation function is applied batch-wise.
        \[f(\mathbf{x}) = \underbrace{\text{sgn}(x_i)\ln(1 + c\,|x_i|)}_{\text{base}} + \underbrace{A e^{-\frac{1}{2}z_i^2} \sin(\omega x_i)}_{\text{perturbation}},
        \quad z_i = \frac{x_i - \mu_{\mathcal{B}}}{\sigma_{\mathcal{B}} + \epsilon}\]
        where \(\mu_B, \sigma_B\) are the batch mean and s.d., \(c = 0.5, A = 0.2, \omega = 2.0, \epsilon = 10^{-6}\) in our experiments.
\end{enumerate}

Specifically, we use the pre-shuffled 100-billion-token sample (\texttt{karpathy/fineweb-edu-100b-shuffle}), distributed as gzip-compressed Parquet shards. We will also adopt NanoChat's data preprocessing pipeline as-is: text is tokenised using a custom Byte-Pair Encodoer with byte fallback (i.e. rare characters are encoded in UTF-8 instead of \<UNK\>), and tokenised sequences are loaded via a distributed data loader that streams directly from the shards during training. No additional filtering or pre-processing is applied beyond the NanoChat pipeline.

\subsection{Evaluation Metrics}

Our primary baseline is the unmodified NanoChat model, which we replicate using the same dataset, processing pipeline, and training hyperparameters to ensure a fair comparison.

%We will also compare against the original OpenAI GPT-2 checkpoint, which achieves a CORE score of 0.2565. The current NanoChat leaderboard record stands at a CORE score of 0.2602, achieved in 2.76 hours on an 8$\times$H100 node \parencite{nanochat}.

We use three complementary metrics to measure both performance and efficiency. First, we will track \textbf{validation loss} (bits per byte) under a fixed compute budget or wall-clock time to measure training efficiency gains. \textbf{``The bits per byte (bpb) metric is a tokenization vocab size-independent metric for evaluating language models. Unlike mean loss, bpb normalizes by the number of bytes that target tokens represent, making it comparable across different tokenizers and vocabulary sizes.''}
\begin{equation*}
    \mathcal{L}_{\text{bpb}} = \frac{\sum_{i=1}^{N} \ell_i \cdot \mathbb{I}[b_i > 0]}{\sum_{i=1}^{N} |b_i|}
\end{equation*}
where
\begin{itemise}
    \item \(\ell_i = -\log P(x_i \mid x_{<i})\) is the per-token cross-entropy loss,
    \item \(b_i\) is the UTF-8 byte length of token \(x_i\),
    \item \(\mathbb{I}[b_i > 0]\) masks out special tokens with zero byte length.
\end{itemise}
\\~\\
Second, to measure absolute performance, we use the CORE evaluation suite from DataComp-LM \parencite{li2025datacomplmsearchgenerationtraining}, which covers 22 diverse tasks. The set of CORE tasks was selected due to their robustness to noise during training, even at small scales. The CORE score is calculated by the mean of \emph{centered accuracies} on these 22 tasks.

Third, we will measure the \textbf{time to GPT-2}, defined as the wall-clock time required for NanoChat 2.0 to surpass GPT-2's CORE score of 0.2565.

\section{Evaluation Metrics}
\subsection{CORE Score}

We reproduce Appenxix G.I from the \textit{DataComp-LM: In search of the next generation of training sets for language models} paper \cite{li2025datacomplmsearchgenerationtraining}, which describes the 22 tasks which make up the CORE score. The CORE score is calculated by the mean of \emph{centered accuracies} on these 22 tasks. The set of CORE tasks was selected due to their robustness to noise during training, even at small scales.

\begin{itemize}

  \item \textbf{AGI Eval LSAT-AR} \cite{zhong2023agieval} (3-shot, 230 examples): Tests model knowledge in the legal domain and evaluates analytical reasoning capabilities.

  \item \textbf{ARC Easy \& ARC Challenge} \cite{clark2018arc} (10-shot; Easy: 2,376 examples, Challenge: 1,172 examples): Four-way multiple choice questions from grade 3--9 science exams. Easy questions require basic science knowledge; challenge questions require procedural reasoning.

  \item \textbf{Big-Bench} \cite{bigbench2023} (10-shot, 6 datasets):
  \begin{itemize}
    \item \textbf{QA Wikidata} (20,321 examples): Models complete factual statements with the correct answer.
    \item \textbf{Dyck Languages} (1,000 examples): Model completes a partially balanced expression of parentheses and braces.
    \item \textbf{Operators} (210 examples): Model is given newly defined operators and computes the output of an expression.
    \item \textbf{Repeat Copy Logic} (32 examples): Requires differentiating instructions from text-to-copy and performing a sequence of operations.
    \item \textbf{CS Algorithms} (1,320 examples): Requires executing algorithms such as recursion and dynamic programming.
    \item \textbf{Language Identification} (10,000 examples): Model identifies the language of a natural language text sequence.
  \end{itemize}

  \item \textbf{BoolQ} \cite{clark2019boolq} (10-shot, 3,270 examples): Binary question answering dataset where the model answers questions about relevant passages.

  \item \textbf{CommonsenseQA} \cite{talmor2019commonsenseqa} (10-shot, 1,221 examples): Five-way multiple choice QA evaluating commonsense knowledge on everyday scenarios.

  \item \textbf{COPA} \cite{roemmele2011copa} (0-shot, 100 examples): Causal reasoning questions where the model selects the more likely of two outcomes using commonsense.

  \item \textbf{CoQA} \cite{reddy2019coqa} (0-shot, 6,304 examples): Conversational QA dataset where the model extracts an answer from a passage given a multi-turn conversation.

  \item \textbf{HellaSwag} \cite{zellers2019hellaswag} (0-shot and 10-shot, 10,042 examples): Four-way multiple choice commonsense reasoning dataset requiring implicit context understanding to select the correct sentence continuation.

  \item \textbf{Jeopardy} \cite{kaggle2019jeopardy} (10-shot, 2,117 examples): Questions posed in the format of the Jeopardy! quiz show, covering a wide variety of topics.

  \item \textbf{LAMBADA} \cite{paperno2016lambada} (0-shot, 5,153 examples): Collection of narratives where the model must predict the final word using full narrative context, not just the local sentence.

  \item \textbf{OpenBookQA} \cite{mihaylov2018openbookqa} (0-shot, 500 examples): Four-way multiple choice QA requiring multi-step reasoning and commonsense knowledge.

  \item \textbf{PIQA} \cite{bisk2020piqa} (10-shot, 1,838 examples): Binary multiple choice QA requiring physical commonsense reasoning.

  \item \textbf{SQuAD} \cite{rajpurkar2016squad} (10-shot, 10,570 examples): QA dataset where the model is given a question and a passage containing the answer.

  \item \textbf{Winograd Schema Challenge} \cite{levesque2012winograd} (0-shot, 273 examples): Binary multiple choice pronoun resolution task requiring commonsense knowledge and contextual understanding.

  \item \textbf{Winogrande} \cite{sakaguchi2020winogrande} (0-shot, 1,267 examples): Extends the Winograd Schema Challenge with a wider variety of domains.

\end{itemize}
